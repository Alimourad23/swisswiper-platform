## What we're building

An internal web app — the heart of SwissWiper performance — that the team logs into to see five areas in one place: Emails (Gmail activity), Calendar (agenda, meeting load), Marketing (LinkedIn/Instagram/TikTok), Sales (pipeline, conversion), Orders (order-to-delivery status). Do not add other frameworks or large libraries without explaining why and getting Ali's OK.

## Modular architecture (non-negotiable)

Every module is a self-contained vertical slice. Enforce this now and going forward:

- Each module owns three places only: its route (`src/app/dashboard/<module>/`), its logic/data layer (`src/lib/<module>/`), and its components (`src/components/<module>/`).
- Modules are registered in ONE place: `src/lib/modules.tsx` (controls the sidebar + live/coming-soon state).
- **Modules NEVER import from each other.** Anything shared goes through the common foundation: auth, the Supabase clients (`src/lib/supabase/*`), design tokens/primitives (`globals.css`, shared components like `Pill`, `ModuleHeader`), and shared services such as notifications (`src/lib/notifications/`).
- Every external data source sits behind a single function (e.g. `getLinkedInMetrics()`, `getInboxView()`, `getTasksData()`). The UI calls the function; swapping the source (export → API) must not change the UI.
- Each module owns its own database tables with its own RLS. Removing a module must not touch another's data.

Design principles: luxury = subtraction; generous white space; restraint over decoration; no clutter.

Language: never "buy/order" → use "commission"; never qualify or discount price.
